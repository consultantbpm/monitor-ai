The user invoked `/ps` to compare a screenshot of the current product state against the project's requirements, and to receive a structured prompt they can use to drive fixes.

## Step 1 — locate the screenshot

If the user provided a path argument, read the image at that path. If they pasted an image directly into the chat, use the image from the conversation. If neither, ask:

> Which screenshot should I analyze? Paste a path or attach the image.

Common screenshot locations for this project (check if not specified):
- `c:\AN\ai-code-exposure-monitor\.teachcm/screenshots/`
- `screenshots/`, `docs/screenshots/`, `assets/screenshots/`
- Clipboard saved files in the project root

## Step 2 — load the requirements

Read project requirements from, in order:
1. `SPEC.md` (canonical, per CLAUDE.md)
2. `BUSINESS_SPEC.md`
3. `CLAUDE.md` itself
4. `.teachcm/notes/` for prior capture about UI/UX

If none of these exist, ask the user where the requirements live.

## Step 3 — compare and report

Produce a structured report with these sections:

### Observed
What the screenshot actually shows. Be concrete: text labels, button positions, colors, states, missing elements.

### Required (per spec)
What the requirements say should be shown at this state. Quote spec lines or paths where possible.

### Non-conformities
A numbered list. For each:
- **What's wrong** — one sentence
- **Severity** — blocker / major / minor / cosmetic
- **Spec reference** — file:line or section name where the requirement lives
- **Suggested fix** — one line, concrete (e.g., *"add `restoreLicense` menu entry to settings drawer, key `R.string.restore_license`"*)

### Conforming
What looks correct (briefly, so the user knows you checked).

## Step 4 — propose a fix prompt

End with a ready-to-use prompt block the user can copy back to you (or to a fresh session) to drive the fixes:

````markdown
## Fix prompt — generated by /ps on <date>

Working in `<project>`. Implement the following changes to bring the UI in line with `SPEC.md`:

1. <non-conformity 1 → concrete change, file paths>
2. <non-conformity 2 → ...>
...

Verify each change against the spec section noted. After implementing, run the standard build (`winstall` or equivalent) and confirm a fresh screenshot resolves the non-conformities.
````

## Do not

- Do not invent requirements not present in the spec files.
- Do not implement the fixes during `/ps` — only diagnose and propose. Implementation comes from the user accepting the fix prompt.
- Do not write to `.teachcm/`. `/ps` is diagnostic, not a learning capture.

## Logging to command history

After producing the diagnosis + fix prompt, append a JSONL line to `c:\AN\ai-code-exposure-monitor\.teachcm/.meta/command-history.jsonl`:

    {"ts":"<ISO8601 now>","cmd":"/ps","arg":"<screenshot path or argument verbatim>"}

Create the file and parent directory if missing.
